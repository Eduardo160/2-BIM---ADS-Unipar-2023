Feature Driven Development

O desenvolvimento a recursos é uma metodologia ágil para desenvolvimento de software orientado a modelos. O FDD foi originalmente idealizado por Jeff de Luca e surgiu em Singapura entre 1997 e 1999, baseado na abordagem Coad (criada por Peter Coad - 1980/1990) e nos processos interativos e enxutos já utilizados por Jeff de Luca. Propõe soluções baseadas no estudo do problema e na compreensão do processo de interação. A abordagem FDD é útil para organizações que precisam desenvolver projetos de software complexos em um curto período de tempo para dar suporte às necessidades mais prioritárias da organização. O FDD se destaca pela ênfase na modelagem de classes e no desenvolvimento de recursos, o que ajuda a manter o sistema organizado e coeso. Embora não seja tão amplamente adotado como alguns outros métodos ágeis, o FDD pode ser uma opção eficaz para projetos que podem ser divididos em características independentes claramente definidas.

Fases do FDD

A FDD é uma metodologia muito objetiva. Possui apenas duas fases; a Concepção & Planejamento que se resume em Pensar um pouco antes de fazer (tipicamente de 1 a 2 semanas) e a construção que é fazer de forma iterativa (tipicamente em iterações de 2 semanas).

Visão Geral da Metodologia FDD

  1. Desenvolvimento por Funcionalidades (Feature-Driven Development): O FDD divide o projeto em funcionalidades menores e gerenciáveis, chamadas de "features". Cada feature representa um componente ou parte do sistema que pode ser desenvolvida.
  2. Lista de Features: A equipe de desenvolvimento cria uma lista de todas as features a serem implementadas no sistema. Cada feature é descrita em detalhes, incluindo sua funcionalidade, requisitos e prioridade.
  3. Projeto de Alto Nível: A equipe realiza um projeto de alto nível para definir a arquitetura geral do sistema. Isso inclui a identificação de objetos de domínio e as relações entre eles.
  4. Desenvolvimento por Feature: As features são atribuídas a equipes ou desenvolvedores individuais. Cada equipe é responsável por projetar, implementar, testar e entregar uma ou mais features específicas.
  5. Modelagem de Classes por Feature: Antes da implementação, a equipe faz uma modelagem de classes detalhada para a feature que estão trabalhando. Isso ajuda a definir as classes e objetos necessários para a feature.
  6. Desenvolvimento Iterativo: As features são desenvolvidas de forma iterativa, com ciclos curtos de desenvolvimento. Isso permite que as equipes entreguem incrementos funcionais do sistema em intervalos regulares.
  7. Inspeção e Revisão: Após a implementação de uma feature, ela é revisada por outras equipes para garantir a qualidade e a conformidade com os requisitos.
  8. Integração Contínua: À medida que as features são desenvolvidas, elas são integradas continuamente ao sistema principal.
  9. Builds Regulares: Builds regulares do sistema são criados e testados para garantir que todas as features se integrem sem problemas.
  10. Entrega Incremental: O sistema é entregue de forma incremental, com cada iteração adicionando novas funcionalidades ao produto.
  11. Teste Completo: Testes abrangentes são realizados em todo o sistema para garantir a qualidade e a funcionalidade.
  12. Manutenção Contínua: Após a entrega inicial, a equipe continua a manter e aprimorar o sistema, adicionando novas features conforme necessário.
